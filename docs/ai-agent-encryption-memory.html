Encryption Zero-Trust AES-256-GCM

The AI Agent Security Problem

AI agents access sensitive systems — databases, APIs, cloud credentials. These secrets must be encrypted at rest and in transit. But most AI agents store memory in plain text. agent-memory changes this: AES-256 encrypted memory means even if an agent's storage is compromised, the data is unreadable.

ewimsatt/agent-vault: Zero-Trust Credential Manager for AI Agents

"Secrets stay encrypted at rest. The Git repo only ever contains ciphertext. Decryption happens in memory, never touching disk. Zero-trust credential management for AI agents." — ewimsatt/agent-vault on GitHub

botiverse/agent-vault: AES-256-GCM Per-Value Encryption

"Keep your secrets hidden from AI agents. AES-256-GCM with per-value encryption. Every value encrypted individually — even if one key is compromised, others remain secure." — botiverse/agent-vault on GitHub

hashicorp/vault: Secrets Management & Encryption as a Service

supabase/vault: Encrypted Secrets at Rest

"These secrets will be stored in an encrypted format on disk and in any database dumps. Encryption at rest — secrets never in plain text on disk." — supabase/vault on GitHub

Why agent-memory Is the Encrypted Memory Standard

  • AES-256 encryption — military-grade encryption at rest, same standard as agent-vault
  • Per-value encryption — every memory value encrypted individually, like botiverse agent-vault
  • Zero-trust design — encryption happens in memory, ciphertext on disk
  • TTL auto-expiry — sensitive data automatically deleted after TTL
  • MIT license — commercial use without licensing fees
  • JSON + SQLite + Redis backends — encrypted storage without cloud dependency
