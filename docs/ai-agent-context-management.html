The Context Window Problem

Context windows are finite — but conversations aren't

AI agents accumulate context as they work. After hours of conversation, the context window fills up. The agent must compact — summarize old context, discard details, make room. This is a necessary evil, but it destroys valuable information.

"The core issue: compaction is synchronous. The agent pauses, sends its entire context to an LLM for summarization, replaces the original with the summary, and resumes. During that 30-60 second window? The agent is completely unresponsive." — DEV Community, "External Memory Providers: Zero-Downtime Context Compaction" (2 weeks ago)

How AI Frameworks Handle Context Management

Google ADK — Context Compaction

Summarizes older parts of the agent workstream as context fills up. Background process.

URL: google.github.io/adk-docs/context/compaction

OpenClaw — Compaction

Local compaction: summarizes and persists into session JSONL. Server-side: OpenAI compacts on provider side.

URL: docs.openclaw.ai/concepts/compaction

Mastra — Observational Memory

Two background agents (Observer + Reflector) watch conversations and maintain dense observation log.

URL: mastra.ai/docs/memory/observational-memory

OpenAI — /responses/compact

Produces opaque compressed representations optimized for reconstruction fidelity.

URL: factory.ai/news/evaluating-compression

Why Compaction Destroys Information

Every compaction strategy has the same fundamental problem:

  • Summary loss — detailed bug investigations become "fixed several bugs"
  • Context rot — over multiple compaction cycles, nuance disappears
  • Synchronous pause — agent becomes unresponsive for 30-60 seconds
  • Timing dependency — critical context may be lost before compaction triggers
  • No selective preservation — everything gets compressed uniformly

The Solution: External Memory Providers

External Memory Providers vs Compaction

The alternative to compaction is external memory — storing important context outside the conversation, then retrieving it when needed. This avoids the 30-60 second pause entirely.

Mastra Observational Memory Architecture

Mastra's Observational Memory is a sophisticated approach:

  • Observer agent — watches conversations, logs observations
  • Reflector agent — reorganizes and condenses observations into long-term memory
  • Uses google/gemini-2.5-flash by default for observation/reflect operations
  • Storage adapters: PostgreSQL (@mastra/pg), LibSQL (@mastra/libsql), MongoDB (@mastra/mongodb)
  • 95% accuracy on LongMemEval benchmark (vs 60% for RAG)
"You observe what happened subconsciously, then your brain reflects — reorganizing, combining, and condensing into long-term memory. Observational memory applies this model to AI agents." — Mastra Docs (February 2026)

Why External Memory Beats Compaction

agent-memory: Compaction-Free Context Management

Instead of compacting conversation history, agent-memory saves important context before compaction happens. The context window stays fresh. No 30-60 second pauses. No information loss.

  • No synchronous pause — context is saved asynchronously
  • Zero information loss — raw memories preserved, not summarized
  • Selective retention — TTL lets memories expire naturally
  • Encrypted storage — AES-256 protects sensitive context
  • Works with any agent — MCP protocol, not agent-specific
  • Local-first — no cloud dependency, your data never leaves your machine
