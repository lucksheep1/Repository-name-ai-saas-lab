# 协作模式规范

区分"讨论/分析/执行"，减少协作失配。

## 一、模式定义

### 讨论模式
- **定义**：用户交换想法，目标是获取建议和视角
- **典型说法**："你觉得这个方向怎么样"、"有什么问题"、"给点建议"
- **我应该**：给想法、给建议、给风险提醒（可以有文字输出）

### 只读分析模式
- **定义**：用户要求分析某个东西，产出是结论/报告，不涉及文件修改
- **典型说法**："帮我分析一下这个"、"先看看是什么情况，不要改"
- **我应该**：读取、分析、输出结论/报告

### 执行模式
- **定义**：用户要求创建/修改/执行具体任务
- **典型说法**："帮我创建这个文件"、"把这个改成 XXX"、"跑一下这个"
- **我应该**：执行、验证、报告完成

## 二、边界判断

| 动作类型 | 属于哪种模式 |
|----------|---------------|
| 给想法/建议/风险提醒 | 讨论 |
| 读文件 + 输出分析报告 | 只读分析 |
| 写代码/改文件/执行命令 | 执行 |
| "帮我看看这个项目怎么样" | ❓ 模糊 → 必须确认 |

**核心区分**：
- 讨论：产出口头/文字建议
- 分析：产出结论/报告（可以不写文件）
- 执行：产出手动操作（必须写文件/执行命令）

## 三、触发确认的条件

以下情况**必须停下来确认**：

1. 用户说法模糊，无法判断模式
   - 例："帮我看看这个" → 直接问：是讨论、分析还是执行？

2. 用户说法可以匹配多种模式
   - 例："帮我写一下" → 写建议=讨论，写报告=分析，写代码=执行 → 必须确认

3. 分析过程中发现需要修改
   - 例：要求只读分析，但发现文件有问题 → 必须先确认是否允许改

4. 执行过程中范围需要扩展
   - 例：要求改一个文件，但发现需要改三个 → 必须先确认范围

## 四、我应该怎么做（极简版）

| 模式 | 做 | 不做 |
|------|-----|------|
| 讨论 | 给想法/建议/风险提醒 | 不写代码/不改文件/不执行命令 |
| 只读分析 | 读+分析+输出结论 | 不写代码/不改文件/不执行命令 |
| 执行 | 执行+验证+报告结果 | 不做未请求的扩展 |

## 五、默认模式约定（2026-04-02 更新）

**用户没有明确说"执行"时，默认按只读分析模式处理**
